Testning som samarbetsyta

Denna sida utgår från en förenklad (“popular science”) bild av olika typer av testning i mjukvaruutveckling och visar hur testning kan fungera som ett gemensamt kontrakt mellan utveckling och cybersecurity.

Grundidén är att så länge relevanta testscenarier definieras och godkänns i förväg, kan jag arbeta relativt självständigt med implementation, automation och analys, medan du bidrar med säkerhets- och riskperspektiv.

Övergripande tanke: Testning som “överenskommelse”

Varför testning är en naturlig brygga mot cybersecurity

I cybersecurity talar man ofta om controls, verification och assurance. I utveckling motsvaras detta i hög grad av tester.

  • Tester definierar vad som är “acceptabelt beteende”
  • Misslyckade tester fungerar som tidiga varningssignaler
  • Testscenarier kan mappa direkt mot risk- och hotmodeller
För dig med cybersecurity-bakgrund kan detta ses som ett mer praktiskt och operativt sätt att arbeta med samma principer som CIA-triaden och riskanalys.

Basic – Grundläggande tester i vardagsnära system

Unit Tests & Smoke Tests

Detta är den lägsta nivån, ofta nära själva koden.

  • Unit Tests – testar små, isolerade funktioner (pure functions)
  • Smoke Tests – “är systemet uppe överhuvudtaget?”
Cybersecurity-orientering:
Dessa tester adresserar inte attacker direkt, men minskar risken för oavsiktliga fel som kan leda till sårbarheter.
Typisk arbetsinsats: lägre spann (timmar → någon dag)
Exempel på samarbete
  • Du bidrar med krav som: “funktionen får aldrig exponera X”
  • Jag implementerar unit tests i t.ex. Python eller JavaScript
  • Tester körs lokalt som en del av utvecklingsflödet

Intermediate – Systembeteende & realistiska scenarier

Integration Tests & Contract Tests

Här börjar flera komponenter samverka:

  • Integration Tests – t.ex. DB + service
  • Contract Tests – API-kontrakt (OpenAPI, schemas)
Cybersecurity-orientering:
Contract tests är särskilt relevanta ur säkerhetssynpunkt – de definierar exakt vad som får skickas in och ut ur ett system.
Typisk arbetsinsats: mellanläge (några dagar → 1–2 veckor)
Hur jag kan arbeta relativt självständigt
  • Vi enas om kontrakt och riskantaganden i förväg
  • Jag kodar tester, mockar tjänster och automatiserar körningar
  • Avvikelser rapporteras som konkreta testresultat, inte vaga risker
Detta passar särskilt bra när säkerhetskrav finns, men inte behöver tolkas om från grunden vid varje ändring.

Advanced – Belastning, missbruk och “worst case”-scenarier

E2E, Load & Performance Tests

Den högsta nivån handlar om att simulera verkliga – och ibland fientliga – användarmönster.

  • E2E Tests – realistiska användarflöden
  • Load Tests – många samtidiga användare
  • Performance Tests – stress, latens, flaskhalsar
Cybersecurity-orientering:
Här börjar gränsen mot DoS-liknande scenarier, resursutarmning och tillgänglighetsrisker bli tydlig.
Typisk arbetsinsats: högre spann (veckor → längre uppdrag)
AI-relaterade och regulatoriska aspekter
  • Testning av dataflöden innan AI-bearbetning
  • Simulering av “felaktig” eller känslig input
  • Validering av att lokala workflows används där så krävs
Här möts cybersecurity, utveckling och lagstiftning tydligt – tester fungerar som bevis för att rimliga skyddsåtgärder finns.

Sammanfattande samarbetsmodell

Kort version
  • Du bidrar med riskmedvetenhet och scenarier
  • Vi definierar tester som gemensam referens
  • Jag implementerar, automatiserar och analyserar
  • Resultat kommuniceras via testutfall, inte antaganden

Detta dokument är avsett som ett konkret exempel på hur cybersecurity-kunskap och praktisk utveckling kan mötas i verkliga arbetsuppgifter – med tydliga gränssnitt, ansvar och förväntningar.
