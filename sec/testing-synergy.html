Samarbete genom testning

Testning som gemensamt språk

Hur fördefinierade tester gör det möjligt att arbeta självständigt men synkroniserat — oavsett om man sitter med koden eller med säkerhetskraven.

Inom professionell mjukvaruutveckling är tester inte bara ett verktyg för att hitta buggar — de är ett samarbetskontrakt. Om två personer kommer överens om vilka tester som ska passera innan arbetet börjar, kan de arbeta relativt oberoende av varandra och ändå veta att resultatet kommer att fungera ihop.

Det här är särskilt kraftfullt i skärningspunkten mellan cybersäkerhet och utveckling: du kan definiera säkerhetskrav som tester, och en utvecklare som Markus kan implementera lösningar som bevisligen uppfyller dem — utan att ni behöver sitta bredvid varandra hela tiden.

Den här sidan förklarar hur det fungerar, med konkreta scenarier indelade i tre nivåer.

Nyckelinsikt — varför testning möjliggör distribuerat samarbete

Tänk dig att du identifierar en säkerhetsrisk: "Systemet ska neka åtkomst efter fem misslyckade inloggningsförsök." Det är ett krav. Om det kravet skrivs som ett test — ett litet program som automatiskt kontrollerar beteendet — kan en utvecklare arbeta helt självständigt tills testet passerar. Ni behöver inte definiera hur det ska lösas, bara vad som ska gälla.

Det kallas ibland contract-based development eller test-driven development (TDD), och det fungerar lika bra med två personer som med tjugo.

Testtyper — en snabbguide

Olika testtyper tjänar olika syften. Här är de vanligaste, med en notering om vilka som är mest relevanta ur ett cybersäkerhetsperspektiv.

Testtyp Vad den verifierar Säkerhetsrelevans
Unit Tests Minsta möjliga kodkomponent — en enskild funktion ★★☆
Integration Att databas + tjänster fungerar tillsammans ★★★
Contract Tests Att API:er uppfyller överenskommen specifikation ★★★
E2E Tests Hela användarflöden, som en riktig användare ★★★
Load Tests Beteende under hög belastning (många samtidiga användare) ★★☆
Smoke Tests Att grundläggande tjänster lever och svarar ★★☆
Security Tests Att kända sårbarhetsmönster inte finns i koden ★★★
Grundnivå

Tydliga kontroller

Enkla, binära tester som verifierar att grundläggande säkerhetsregler följs. Lämpliga som inträde i testbaserat samarbete.

⏱ Typisk arbetsinsats: 2–8 timmar per scenario
🛡️
Input Validation — stoppa farlig indata
Grundläggande skydd mot injection och felaktig data
📖 Orientering — vad menas?

Input validation innebär att kontrollera all data som kommer in i ett system — formulärfält, API-anrop, URL-parametrar — innan den bearbetas. Det är en av de viktigaste säkerhetskontrollerna och relaterar direkt till Integrity i CIA-triaden.

Scenario: Du definierar kravet: "Ingen indata som innehåller SQL-liknande mönster får nå databasen ofiltrerad." Markus skriver sedan unit tests som testar exakt detta — och implementerar lösningen tills alla tester passerar.

// Exempel: Unit test (Chai + Mocha) som verifierar input sanitization
describe('sanitizeInput', () => {
  it('should reject SQL injection patterns', () => {
    const malicious = "'; DROP TABLE users; --";
    expect(sanitize(malicious)).to.not.include("DROP");
  });
});

Din roll: Definiera vilka typer av farlig indata som ska blockeras, baserat på dina kunskaper om vanliga attack vectors (från kursens Module 2). Markus roll: Implementera valideringen och skriva tester med Chai och Mocha (som han har certifierad erfarenhet av) som bevisar att kraven uppfylls.

Password Policy Enforcement
Testa att lösenordskrav verkligen upprätthålls

Unit tests testar den allra minsta kodkomponenten — en enda funktion. De är snabba (millisekunder), körs ofta, och ger omedelbar feedback. Perfekt för att verifiera regler som "lösenord måste vara minst 12 tecken" eller "måste innehålla minst en siffra".

Scenario: Utifrån bästa praxis (som du stött på via Security Controls i Module 3) sätter du kraven: minimilängd, komplexitet, att lösenord inte lagras i klartext. Markus skriver enhetstester för varje krav.

Security Headers — Smoke Tests
Verifiera att grundläggande skydd alltid är aktivt
📖 Orientering

Smoke tests är enkla "lever det?"-kontroller. De verifierar inte komplex logik, utan att grundläggande saker är på plats. Att ett API svarar. Att en server har rätt security headers. De är billiga att skriva och enormt värdefulla.

Scenario: Du listar vilka HTTP-headers som måste finnas (X-Content-Type-Options, Strict-Transport-Security, Content-Security-Policy). Markus sätter upp ett smoke test som körs dagligen och larmar om något saknas.

↗ Samarbetsmodell

Din roll: Ta fram checklistan utifrån ramverk som NIST eller OWASP. Markus roll: Implementera HelmetJS-konfiguration och skriva automatiserade smoke tests — antingen med Node.js eller som Python-skript som gör HTTP-anrop och verifierar headers. Ett kompakt men professionellt samarbetsstycke.

Mellannivå

Sammansatta flöden

Tester som involverar flera systemdelar och realistiska användarscenarier. Kräver mer planering men speglar professionell verklighet.

⏱ Typisk arbetsinsats: 8–30 timmar per scenario
🔐
Authentication Flow — E2E Test
Testa hela inloggningsflödet som en verklig användare
📖 Orientering

End-to-End (E2E) tests simulerar en riktig användare: de öppnar en webbsida, fyller i fält, klickar på knappar, och verifierar vad som händer. Det är det mest realistiska testet — men också det långsammaste och dyraste att underhålla. E2E-tester skrivs med verktyg som Cypress eller Playwright.

Scenario: Ni definierar tillsammans hela authentication flow: registrering → verifiering → inloggning → session → utloggning. Ni definierar vad som ska hända vid felaktigt lösenord, utgången session, och vid brute-force-försök (t.ex. rate limiting efter 5 försök).

↗ Samarbetsmodell

Din roll: Utifrån CISSP-domänen Identity and Access Management sätter du kravspecifikationen: vilka beteenden ska och inte ska vara möjliga. Markus roll: Bygga hela autentiseringsflödet med Passport.js, JWT, och Express middleware, samt E2E-tester som verifierar varje definierat scenario. Ni kan arbeta i veckor utan daglig synk — testerna garanterar att kontraktet hålls.

Passport.js JWT IAM E2E Test Kontrakt
📄
API Contract Testing
Garantera att gränssnitt mellan system håller sina löften
📖 Orientering

Contract tests verifierar att ett API levererar exakt den data det lovat — rätt fält, rätt format, rätt statuskoder. Om ett API lovar att returnera { "user": { "id": 1, "role": "admin" } }, så testar ett contract test att det faktiskt ser ut så. Specifikationer som OpenAPI (Swagger) gör detta maskinläsbart.

Scenario: Ni designar tillsammans ett API för incidentrapportering. Du specificerar vilka fält som krävs (severity, timestamp, affected_asset, reporter_id), vilka roller som får göra vad, och vilka HTTP-statuskoder som ska returneras vid olika fel.

↗ Samarbetsmodell

Det här är det kanske mest naturliga samarbetsmönstret: du specificerar kontraktet (baserat på domänkunskap från säkerhetsfältet), och Markus implementerar API:et med Node.js/Express och skriver contract tests som bevakar att specifikationen hålls. Om en tredje person senare ändrar API:et, bryter testet — och alla vet direkt.

REST API Express OpenAPI Contract Test
🔗
Integration Tests — databas + behörighet
Verifierar att flera systemdelar samverkar korrekt
📖 Orientering

Integration tests ligger mellan unit tests och E2E tests. De testar att två eller fler komponenter fungerar ihop — typiskt en databas och en tjänst, eller en autentiseringsmodul och ett API. De är snabbare än E2E men fångar fler verkliga problem än enhetstester.

Scenario: Ni vill verifiera att en användare med rollen "analyst" kan läsa men inte skriva säkerhetsloggar, medan rollen "admin" kan göra båda. Det kräver att databasen, autentiseringen och API-lagret alla samverkar korrekt.

↗ Samarbetsmodell

Din roll: Designa rollmatrisen — vilka roller finns, vad får de göra? (CISSP: Identity and Access Management.) Markus roll: Implementera RBAC i databas + API och skriva integrationstester med PostgreSQL som verifierar att varje kombination av roll × åtgärd beter sig som specificerat. Med hans erfarenhet av SQL, Express och Passport.js hänger alla bitarna ihop.

PostgreSQL RBAC Express Integration Test
Avancerad nivå

Systemövergripande & AI-drivet

Komplexa scenarier som involverar automatisering, maskininlärning och realtidsövervakning. Representerar professionellt arbete nära produktionsmiljöer.

⏱ Typisk arbetsinsats: 30–120+ timmar per scenario
🔄
Automated Security Regression Suite
En testsvit som bevakar alla kända sårbarheter — automatiskt
📖 Orientering

En regression suite är en samling tester som körs automatiskt vid varje kodändring. Tanken: varje gång ett säkerhetsproblem upptäcks och fixas, skrivs ett test som garanterar att det aldrig kommer tillbaka. Sviten växer över tid och blir organisationens säkerhetsminne.

Scenario: Ni bygger en levande testsvit som täcker OWASP Top 10: injection, broken authentication, sensitive data exposure, XSS, m.fl. Varje gång en ny sårbarhet rapporteras i branschen, lägger ni till ett test.

↗ Samarbetsmodell

Din roll: Omvärldsbevaka — vilka nya attack vectors dyker upp? Vilka bör testas? Du håller svitens "kravkatalog" uppdaterad. Markus roll: Bygga testramverket, implementera varje nytt test, och sätta upp CI/CD-integration så att sviten körs automatiskt. Ett pågående samarbete snarare än ett enskilt projekt — med tydlig arbetsdelning och automatiserad feedback.

OWASP Top 10 CI/CD Regression Suite Pågående
🤖
ML-driven Anomaly Detection Pipeline
Maskininlärning som identifierar avvikande beteenden i loggar
📖 Orientering

Anomaly detection med maskininlärning innebär att en modell lär sig vad "normalt" beteende ser ut — och larmar när något avviker. Det kräver att man definierar vad som är normalt, vilken data modellen ska tränas på, och vilken false positive rate som är acceptabel. Det sista är ett säkerhetsbeslut, inte ett tekniskt sådant.

Scenario: Ni bygger en pipeline som tar in loggdata (nätverkstrafik, inloggningsförsök, API-anrop), tränar en modell att känna igen mönster, och flaggar avvikelser. Hela arbetet kan ske lokalt om datan är känslig.

↗ Samarbetsmodell

Din roll: Definiera vilken typ av avvikelser som är intressanta, vilken false positive-nivå som är acceptabel, och vilken data som är säker att använda (med hänsyn till GDPR och eventuell anonymisering). Markus roll: Bygga pipeline i Python med Pandas för databearbetning, Scikit-learn eller TensorFlow för modellträning, och automatiserade tester som verifierar att modellen uppfyller överenskomna precision/recall-trösklar. Hans medvetenhet om datasanitisering och lokala arbetsflöden gör det möjligt att hantera känslig data utan att den lämnar den egna miljön.

TensorFlow Scikit-learn Pandas GDPR Lokal pipeline

Load & Resilience Testing under attacksimulering
Hur beter sig systemet under simulerad belastning och attack?
📖 Orientering

Load tests simulerar hög trafik — men i en säkerhetskontext handlar det inte bara om prestanda utan om resilience: klarar systemet en DDoS-liknande belastning? Fortsätter rate limiting att fungera? Degraderas säkerhetsfunktioner under stress?

Scenario: Ni simulerar tusentals samtidiga inloggningsförsök mot ett system. Ni definierar i förväg: "Systemet ska aldrig ge åtkomst till fler än X samtiliga sessioner per användare", "Svarstiden ska inte överstiga 2 sekunder", "Rate limiting ska aktiveras efter Y anrop/minut".

↗ Samarbetsmodell

Din roll: Definiera tröskelvärden och acceptabla beteenden utifrån Availability-aspekten av CIA-triaden och din förståelse för DDoS/brute-force-attackmönster. Markus roll: Skriva lastgenereringsscripts (Python eller Node.js), implementera rate limiting middleware, och bygga performance tests med definierade trösklar. Visualisering av resultat med D3.js eller Matplotlib kan ge tydliga rapporter att dela med beslutsfattare.

Python Node.js D3.js / Matplotlib DDoS Load Test

Sammanfattning: Testbaserat samarbete handlar inte om att kunna allt själv. Det handlar om att kunna definiera vad som ska gälla (din domänkunskap inom säkerhet) och lita på att implementationen verifieras automatiskt (utvecklarens tester). Ju tydligare kraven formuleras som tester, desto mer oberoende kan varje person arbeta — och desto högre kvalitet levereras.
